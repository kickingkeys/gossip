Session Model (Privacy Fix)

  • Group session: one long-running session for #welcome channel
  • DM sessions: one per member, fully isolated
  • DM session context loaded from DB at start: summary + DM history + chat + donny_memory
  • Person A's DM content is never visible when Donny responds to Person B
  • Proactive DMs sent via cron session, reply arrives in the member's DM session
  • Handoff works because dm_history + donny_memory are in shared DB

Proactive Outreach (Cron, not Heartbeat)

  • Runs as cron job every 30 min in isolated session (NOT heartbeat)
  • Code-level pre-check runs BEFORE any LLM call:
  •   1. Quiet hours? → skip entirely, $0
  •   2. Anyone active since last check? → no → skip, $0
  •   3. Anyone quiet 4h+? → no, chat not idle 3h+? → skip, $0
  •   4. Only if pre-check passes → call Sonnet to compose message
  • Result: ~80% of checks cost $0 (just DB queries)
  • Rate limit: max 2-3 proactive DMs per person per day

Donny's Memory (SQLite Table)

  • Stored in donny_memory table in SQLite (NOT a markdown file)
  • Schema: (id, timestamp, channel_type, target, content)
  • channel_type: "group" | "dm/{member_name}" | "proactive/{member_name}"
  • Read: SELECT last 30 rows ORDER BY timestamp DESC
  • Write: INSERT after every response
  • Trim: DELETE WHERE timestamp < now - 48h (on every write)
  • Concurrent-safe — SQLite WAL handles simultaneous reads/writes
  • No file corruption risk from parallel sessions

Flow 1: Group Chat Response (@mention / "donny" keyword)
1
Discord Message arrives in #welcome channel
2
Gateway Log message to data/chat/YYYY-MM-DD.md • update chat_activity in DB • identity resolution (4-layer cascade)
3
Gateway Check trigger: contains @donny or "donny"? → route to group session
4
Donny Single tool call: gossip_context(type="group") → returns all summaries + last 60 msgs + donny_memory + gossip history
5
Donny Generate response (Sonnet) → 1-2 sentences, one line
6
Data INSERT donny_memory (channel: "group") • INSERT gossip_history if idle drop
Single tool call, not four. gossip_context(type="group") returns everything Donny needs in one response. One LLM turn for context loading + one for generation = fast reply. SOUL.md + skill are in the system prompt (cached).
Flow 2: DM Conversation (someone DMs Donny)
1
Discord DM arrives from a member
2
Gateway Identity resolution • INSERT dm_history (direction: inbound) • route to per-user DM session
3
Donny Single tool call: gossip_context(type="dm", member="surya") → their summary + DM history + last 20 chat msgs + condensed others + donny_memory
4
Donny Generate response (Sonnet) → natural, warm, friend tone
5
Data INSERT donny_memory (channel: "dm/surya") • INSERT dm_history (direction: outbound)
Privacy fix: Each member has their own DM session. Person A's DM content is NEVER in context when Donny responds to Person B. Donny knows what it said to everyone (via donny_memory table), but not what others said privately. Chat context (last 20 msgs) lets Donny reference group activity: "lol did you see what ryan said earlier."
Flow 3: Proactive Outreach (Cron, every 30 min)
1
Cron Proactive check cron fires (isolated session)
2
Pre-Check Code-level gate (no LLM): quiet hours? → skip. Nobody quiet 4h+? → skip. Chat not idle 3h+? → skip. No new data? → skip.
3
Pre-Check Check proactive DM counts today per member (from donny_memory table): anyone at limit (3/day)? → remove from candidates
4
Donny Only if pre-check passes: gossip_context(type="proactive") → all summaries + donny_memory + chat activity
5a
Donny If someone needs a DM: compose message → send_dm tool → INSERT dm_history (outbound) + INSERT donny_memory
5b
Donny If chat idle 3h+: compose idle drop → send to group → INSERT gossip_history + INSERT donny_memory
Cost savings: ~80% of cron runs hit the pre-check gate and stop without calling the LLM. Quiet hours (11pm-9am) = 0 LLM calls. Calm evenings where nobody's quiet for 4h = 0 LLM calls. Only interesting state changes trigger Sonnet.
Flow 4: Background Synthesis (Cron, every 2h per member, staggered)
1
Cron Isolated session spins up for member X
2
Email Filter Code-level pre-filter: fetch emails → drop medical, financial, legal, romantic (domain + subject pattern matching) → pass only social/travel/event emails
3
Google Fetch calendar events (7d ahead + 7d back) • use pre-filtered email snippets
4
Data Read: chat log mentions of member X (48h) + previous summary (for continuity + Donny Notes carry-forward)
5
Haiku Synthesize structured summary: This Week, Patterns, Relationships, Recent, Flagged, Donny Notes
6
Validate Check output has all required sections • if malformed, retry once with stricter prompt • if still bad, keep previous summary
7
Data Write to data/summaries/{member}.md • update sync_state in DB
Privacy safeguards: (1) Email pre-filter runs in code, not LLM — medical/financial/legal emails never reach Haiku. (2) No DM history fed to synthesizer — DM content stays in DM sessions only, loaded from dm_history table. (3) Validation ensures summary format is consistent for main agent to parse.

Identity Resolution — 4-Layer Cascade

All layers case-insensitive. First match wins. Always backfill discord_id on success.

1
discord_id (exact numeric match)
Most reliable. Set on first message or backfilled from guild member list.
↓ miss
2
discord_username (case-insensitive)
What they entered during onboarding. Also checks global_name from Discord.
↓ miss
3
nicknames (case-insensitive, comma-separated)
NEW field. Checked against Discord username AND global_name. "surya, sk, kickingkeys"
↓ miss
4
display_name (case-insensitive fuzzy)
Last resort. Checks against Discord global_name and display_name.
Always backfill: If resolution succeeds at any layer and discord_id is still NULL, immediately UPDATE the members table. Next time, layer 1 catches it. One function: resolve_member() handles all layers — _find_member() merged and deleted.

Current Bugs (v1)

  • Case-sensitive matching — "KickingKeys" vs "kickingkeys" fails
  • No nicknames field — only username and display_name
  • discord_id often NULL — portal onboarding doesn't capture it
  • Two resolution functions — resolve_member() and _find_member() diverge
  • No global_name check — Discord users have username AND global_name
  • Silent failure — returns None, bot ignores the person entirely

Fixes (v2.1)

  • All string comparisons use LOWER() in SQL + .lower() in Python
  • Add nicknames TEXT column to members table
  • Add nicknames field to invite.html onboarding form
  • Merge _find_member() into resolve_member()
  • Check Discord global_name at layers 2 and 4
  • Always backfill discord_id on first match
  • On None result: log warning with Discord user details for debugging
Google OAuth "Testing" mode: If the OAuth app is not published/verified, refresh tokens expire after 7 days. Every member's Google connection silently dies after a week. Detection fix: when sync gets a 401, mark Google as disconnected in sync_state, DM the member to reconnect. Long-term fix: publish the OAuth app (requires Google review, privacy policy, homepage).

Edge Cases, Fixes & Safeguards

Every identified failure mode and how the architecture addresses it.

# Issue Severity Fix in v2.1 Status
1 DM privacy leak — Person A's DMs visible when responding to Person B if sessions are shared Critical Per-user DM sessions (isolated). Context loaded from shared DB (dm_history, donny_memory), never from another session's conversation history. FIXED
2 Heartbeat cost — Every 30min heartbeat calls Sonnet even when nothing to do ($1+/day wasted) Critical Replaced heartbeat with cron + code-level pre-check gate. DB queries check quiet hours, activity timestamps, DM counts. LLM only called when pre-check passes (~20% of the time). FIXED
3 OAuth testing mode — Google tokens expire after 7 days if app not published Critical Detection: when sync gets 401, mark Google disconnected in sync_state, DM member to reconnect. Long-term: publish OAuth app. FIXED
5 Cold start — New member has no summary for up to 2h after onboarding Critical Trigger immediate synthesizer run after deep sync completes (in same background thread). Summary ready within 1-2 min of OAuth connect. FIXED
6 Context latency — 3-4 tool calls to load context = 5-15s delay before reply Significant Single tool call: gossip_context(type, member) returns everything pre-assembled. One LLM turn for loading instead of four. FIXED
7 Memory file corruption — Simultaneous writes to donny_memory.md from parallel sessions Significant Moved donny_memory to SQLite table. WAL mode handles concurrent reads/writes safely. No file-level race conditions. FIXED
8 Synthesizer quality — Haiku may produce inconsistent summary formats Significant Post-synthesis validation: check all required sections exist. If malformed, retry once with stricter prompt. If still bad, keep previous summary. Previous summary fed as template. FIXED
9 Email sensitivity — Medical/financial/legal emails leaking into summaries Significant Code-level pre-filter in email_filter.py: pattern-match domains and subjects, drop sensitive categories BEFORE data reaches Haiku. Not relying on LLM for privacy filtering. FIXED
10 DM content in summaries — Synthesizer might include private DM statements in summaries visible to group Significant DM history completely removed from synthesizer input. Synthesizers only see: calendar, pre-filtered email, chat mentions, previous summary. DM content stays in DM sessions only. FIXED
11 Proactive DM handoff — Cron composes DM, but member's reply arrives in different DM session with no context of what was sent Moderate send_dm tool writes to BOTH donny_memory table AND dm_history table. DM session loads both at start, so it sees what Donny sent. FIXED
12 Chat windowing — Parsing "last 60 messages" across day-file boundaries, multi-line messages breaking line count Moderate Parse by [HH:MM] timestamp pattern as message boundary. Read files from today backwards until 60 messages collected. Handles multi-line and day gaps. FIXED
13 Discovery spam — Bot DMs undiscovered users every 12h until they join or block Moderate New discovery_log table: (discord_id, attempted_at, success). Skip anyone contacted in last 7 days. Max 1 DM per undiscovered user per week. FIXED
14 Running jokes lost — donny_memory trims after 48h, losing inside jokes and relationship dynamics Moderate "Donny Notes" section in member summaries. Synthesizer carries forward from previous summary. Running jokes, tone preferences, things to avoid persist indefinitely. FIXED
15 Portal spy language — connect_sources.html says "intel" and "gossip engine" breaking the friend illusion Moderate Rewrite: "intel" → "some context about you". Delete "gossip engine" entirely. Keep it casual — "help donny get to know you." FIXED
Remaining Watch Items

Prompt Cache Consistency

  • SOUL.md + skill must be at exact same position in system prompt every call
  • If OpenClaw reorders system prompt sections, cache misses and cost doubles
  • Monitor: log cache_read_input_tokens from API response
  • If cache hit rate drops below 80%, investigate prompt ordering

Everyone Goes Quiet

  • If all members go silent, pre-check gate keeps passing → Donny DMs one person per cycle
  • Could burn through daily DM budget (3/person) in a few hours
  • Safeguard: global cap of 8-10 proactive DMs total per day across all members
  • After global cap hit, only send DMs in response to inbound messages

Synthesizer Staleness

  • Summaries update every 2h — calendar changes between runs are missed
  • "Play dumb" approach mitigates: Donny asks "what are you up to?" not "I see you added dinner"
  • Watch for: Donny referencing stale plans that changed
  • If frequent issue: add event-driven synth trigger when calendar webhook fires

Discord Rate Limits

  • Burst DMs (proactive cron sending 3+ DMs quickly) can hit Discord limits
  • OpenClaw's Discord adapter likely handles this internally
  • Safeguard: stagger proactive DMs by 30s minimum between sends
  • Custom send_dm tool should include sleep(30) between calls
